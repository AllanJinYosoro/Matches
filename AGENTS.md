# AGENTS.md

## 不可违反的边界

- 不把EXE、DLL、数据库、专有资源、凭据或个人数据提交到 Git。
- 需要样本时，创建最小、脱敏、可公开的测试夹具。
- 不声称反编译结果是“原始源码”；函数名、类型和控制流可能不完整或不准确。
- 只为明确的兼容或功能目标做定向分析，不进行无目标的整包反编译。
- 若任务涉及绕过授权、账号系统、安全机制或分发第三方专有代码，先停止并向用户说明。

## 默认技术策略

1. 先观察并记录用户可见行为、输入、输出和边界情况。
2. 优先使用 Everything 官方 IPC/SDK 或公开接口重新实现本地搜索。
3. 优先写独立的新代码；只有无法从行为确认时，才定向分析对应二进制函数。
4. 明文 Web 资源可用于理解界面和 JS↔原生协议，但不要整段复制专有实现。
5. 默认不复现已失效或非必要的账号、聊天和在线服务。
6. 在技术栈尚未由首个功能决定前，不创建框架、插件系统或通用抽象。

## 每项功能的工作流

1. 在修改前读完相关文件，并搜索所有调用方。
2. 写清要保留、改变和删除的行为；不确定项标为待确认。
3. 找到能满足需求的最小实现。
4. 为非平凡逻辑留下一个最小可运行检查。
5. 与参考实现做只读对照，记录有意差异。
6. 运行最小相关验证，并检查 `git diff` 与 `git status`。

## 实现约定

- 目标平台默认为 Windows。
- 文本文件使用 UTF-8；避免无关的编码和换行符变化。
- 优先标准库、Windows 原生能力和仓库已有代码。
- 不为未来需求提前增加依赖、配置层、接口、工厂或插件框架。
- 修复根因所在的共享路径，不在每个调用点重复打补丁。
- 涉及物理设备、性能上限或有意简化时，用 `ponytail:` 注释写明上限及升级条件。
- 不修改仓库外文件，除非用户在当前任务中明确授权。

## 开始任务前

- 阅读本文件。
- 运行 `git status --short`。
- 确认任务属于“复现、改变、删减”中的哪一种。
- 若需要参考火柴，只读检查最小必要文件，并避免输出敏感内容。
